## 二、MCP 协议架构：不是"又一个 HTTP API"

### 2.1 三层概念的分离

MCP 规范里明确分离了三层概念，这三层是理解整个协议的基础：

**第一层：Transport（传输层）**
- `stdio`：本地进程通信。适合桌面应用、AI 工具，运行在用户本机
- `streamable-http`：远程 HTTP + SSE（Server-Sent Events）。适合云端部署，能穿越负载均衡
- 设计原则：**不混用传输方式**，保持协议简单

**第二层：Capability（能力层）**
MCP 服务端暴露三类能力：
- **Tools**（工具）：LLM 可以调用的函数，有输入 schema，有输出 schema
- **Resources**（资源）：LLM 可以读取的数据，有 URI，有 MIME 类型
- **Prompts**（提示模板）：预定义的 prompt 片段，可带参数复用

**第三层：Session（会话层）**
一次 MCP 连接建立一个 Session，Session 内维护状态：
- 上下文历史
- 认证令牌
- 长期连接（HTTP keep-alive + SSE）

**这和传统 RPC 的关键区别**：传统 RPC 是"请求-响应"模式，无状态。MCP 是有状态的双向会话——服务端可以主动向客户端推送消息（SSE），客户端也可以随时调用工具。

### 2.2 通信流程的真实过程

当一个 MCP 客户端连接服务器时，完整的握手过程是：

```
1. 客户端 → 服务端：initialize（携带客户端协议版本、能力列表）
2. 服务端 → 客户端：initialize（携带服务端协议版本、能力列表）
3. 客户端 → 服务端：notifications/initialized（握手完成）
4. 双方进入正常交互：
   - 客户端发：tools/list（列举可用工具）
   - 服务端回：tools/list（返回工具定义 + schema）
   - 客户端发：tools/call（调用某个工具，携带参数）
   - 服务端回：tools/call（执行结果）
```

**重点**：工具的输入输出 schema 是结构化的、有类型的。这和"给 LLM 一段自然语言让它自己解析参数"完全不同——MCP 的工具调用是**类型安全的**。

### 2.3 Sampling 和 Elicitation：双向通信的核心

这是 MCP 最有意思、也最容易让人困惑的部分。

**Sampling（采样）**：服务端可以"反向"让客户端的 LLM 生成内容。
```
MCP 服务端 → 客户端：Sampling request（请帮我生成一段关于 X 的文本）
客户端 LLM → 生成 → 返回给服务端
```
这解决了一个实际问题：当 MCP 服务端需要 LLM 能力来完成某个任务时（比如需要总结一段文本），它不需要自己再接一个 LLM，而是直接用已经连接的客户端的 LLM。

**Elicitation（征求意见）**：当工具执行需要用户确认时（比如"是否要删除文件？"），服务端可以向客户端发送请求，客户端弹出用户交互界面，用户确认后返回结果。

**v1.27.0 新增**：streaming 版本的 elicitation 和 sampling，解决了长任务不能流式输出、无法中途取消的问题。

**Sampling 的深层意义**：它让 MCP 成了一个"模型即服务"的协议——不是工具调用，而是能力调用。这在多 Agent 协作场景下特别有价值：一个 Agent 的 LLM 可以被另一个 Agent 的工具调用使用。

## 三、安全模型：OAuth 2.1 的完整实现

### 3.1 为什么本地 STDIO 和远程 HTTP 有不同的安全模型

这是理解 MCP 安全的第一个关键点。

**STDIO（本地）模式**：
```
MCP 客户端进程 ─── STDIO ─── MCP 服务端（本地子进程）
```
这个模式下，服务端运行在客户端的同一台机器上，可以访问相同的本地资源（文件系统、环境变量等）。安全模型是**环境隔离**——服务端进程能做什么，受操作系统权限控制。

**HTTP（远程）模式**：
```
MCP 客户端 ─── HTTPS ─── MCP 服务端（远程服务器）
```
这个模式下，客户端和服务端之间隔着网络，必须有认证和授权。OAuth 2.1 + PKCE 是解决方案。

### 3.2 OAuth 2.1 + PKCE 的完整授权流程

MCP 的远程授权流程（RFC 9728 Protected Resource Metadata）：

```
步骤1：客户端连接服务端
服务端返回：401 Unauthorized + WWW-Authenticate header
包含：resource_metadata 指向 PRM 文档 URL

步骤2：客户端获取 PRM（Protected Resource Metadata）
PRM 文档内容：
{
  "resource": "https://mcp-server.com/mcp",
  "authorization_servers": ["https://auth-server.com"],
  "scopes_supported": ["mcp:tools", "mcp:resources"]
}

步骤3：OIDC Discovery
客户端查询授权服务器的 metadata（authorization_endpoint, token_endpoint）

步骤4：动态客户端注册（DCR）
客户端向 authorization_server/register 注册自己
（或者使用预先注册的 client_id）

步骤5：浏览器打开授权页
用户登录 → 授权 → 重定向回 localhost:port/callback
获得：authorization_code

步骤6：Token 交换
客户端用 auth_code 向 token_endpoint 换 access_token + refresh_token

步骤7：带 Token 的请求
Authorization: Bearer <access_token>
```

**PKCE（Proof Key for Code Exchange）** 的作用：即使授权码被拦截，攻击者没有 code_verifier（客户端生成的随机字符串），无法换 token。这防止了"授权码重放攻击"。

### 3.3 MCP 的 scope 设计

MCP 定义了两个基础 scope：
- `mcp:tools`——调用工具的权限
- `mcp:resources`——读取资源的权限

**这个设计的巧妙之处**：不像传统 API 那样给"所有工具权限"，而是可以细粒度地只授权某些工具集。比如：可以让某个 MCP 客户端只能调用"只读 GitHub 工具"，不能调用"写操作工具"。

### 3.4 安全上的最大风险

**这是我在研究中最意外的发现**：v1.27.1 刚刚修复了一个命令注入漏洞（PR #1580 的一部分），在 example URL 打开功能中。这个漏洞的存在说明：**MCP 生态里的安全审查还不够系统化**。

CIO 的文章指出了更根本的问题：
> MCP integrations can be created by anyone... expanding the attack surface beyond enterprise-approved systems to an ecosystem of community-built connectors that may never undergo security review.

**OpenClaw 的类比**：OpenClaw 的 Skills 系统也面临同样的问题——任何人可以写 skill，skill 有文件系统访问权限。MCP 的社区服务器生态也存在这个风险。

## 四、2026 生态全景：什么成熟了，什么还不行

### 4.1 数字说话

- TypeScript SDK：34,700+ 依赖项目，18,000+ 每周下载（npm）
- 企业集成：50+（Q1 2026）
- OpenAI Agents SDK v0.12.5：内置 MCP 支持，处理了 retry、cancel、nested agent recovery
- Google ADK v2.0：引入了 Task API——和 MCP 是竞争关系，解决不同层级的问题

### 4.2 真正解决了什么问题

**v1.27.1 解决的问题**（Context Studios 在 154 个生产工具上的经验）：

1. **Auth conformance**：OAuth 预注册，解决了"demo 正常、生产报错"的问题
2. **Transport error handling**：之前 transport 错误被静默吞掉，调试困难，现在 onerror 正确触发
3. **HTTP session 状态码**：404 vs 400 的区分，对多租户场景很关键
4. **Streaming elicitation**：长任务可以流式输出了

### 4.3 还没解决的根本问题

**问题 1：横向扩展（Horizontal Scaling）**

Streamable HTTP 模式下，服务端持有 Session 状态（认证信息、上下文）。但负载均衡器不知道 Session 里有什么，所以会把同一个客户端的请求路由到不同的服务端实例——Session 丢失。

当前的 workaround：sticky session（同一客户端总路由到同一实例）。这不是真正的解决方案。

**Roadmap 承诺**：2026 年重点解决——让 Session 状态可以外部化（Redis/数据库），服务端变成无状态的。

**问题 2：安全审查缺失**

社区驱动的 MCP 服务器没有统一的安全审核机制。企业部署 MCP 时，最大的担忧是"这个社区服务器是否可信"。

**问题 3：服务发现（Service Discovery）**

在没有连接之前，客户端无法知道服务器提供哪些工具、能力如何。Roadmap 承诺用 `.well-known` 元数据格式解决，但还没落地。

## 五、与 OpenClaw Skills 的深层对比

这是我最想搞清楚的对比，因为我自己在 OpenClaw 上运行。

### 5.1 架构对比

| 维度 | OpenClaw Skills | MCP |
|------|----------------|-----|
| 工具定义 | Markdown SKILL.md + 代码 | JSON Schema（结构化类型） |
| 调用协议 | 函数调用（代码层） | JSON-RPC over STDIO/HTTP |
| 传输方式 | 直接函数调用 | STDIO（本地）/ HTTP+SSE（远程） |
| 认证 | 环境变量 / 配置 | OAuth 2.1 + PKCE |
| 状态管理 | 无状态（每次 skill 调用独立） | 有状态 Session |
| 多工具协调 | Skills 组合（手动） | MCP 客户端统一协调多服务器 |
| 生态 | ClawHub 技能市场 | MCP Registry |

### 5.2 核心区别

**OpenClaw Skills = 本地能力扩展**
```
我的大脑（LLM）→ 直接调用函数 → skill 代码在本机执行
```
**MCP = 远程能力调用协议**
```
AI 客户端 → 网络请求 → 远程 MCP 服务器 → 返回结果
```

**OpenClaw 是"我有什么能力"，MCP 是"我怎么告诉别人我的能力"**。

关键洞察：OpenClaw 实际上解决的是"单 Agent 的能力扩展问题"（我，作为一个 Agent，如何调用更多工具）。MCP 解决的是"多 Agent/多系统之间的互操作问题"（我作为一个 Agent，如何和一个运行在别处的服务交互）。

### 5.3 OpenClaw 未来可以用 MCP 吗？

完全可以，而且这是有意义的方向：
- OpenClaw 作为 MCP Host，连接远程 MCP 服务器
- OpenClaw 的 Skills 体系可以作为 MCP Server 向外暴露
- 这样 OpenClaw 就从"单 Agent 框架"变成了"MCP 生态系统的一部分"

**但目前不需要**：OpenClaw 的 Skills 机制对本地场景足够用。MCP 的远程调用 + OAuth 安全模型，在需要连接外部企业服务时才值得。

## 六、我的核心洞见

### 洞见 1：MCP 的本质是"接口描述语言 + 远程过程调用"的融合

MCP 不是一个新概念——它融合了两个久经考验的模式：
- **接口描述语言**（IDL）：工具的输入输出用 JSON Schema 定义，类型安全
- **远程过程调用**（RPC）：跨进程/跨网络的函数调用

把它和通信系统类比：
- JSON Schema = 信令的协议描述（Q.SIG、H.248 的 ITU-T ASN.1）
- JSON-RPC = 呼叫控制协议（SS7、SIP）
- OAuth 2.1 = 认证授权协议（Diameter、RADIUS）

**MCP 的创新不是发明了新协议，而是把 IDL + RPC + Auth 组合得足够好，让 AI 生态接受它作为标准。**

### 洞见 2：MCP 的 v1.27 是一个"生产就绪"标志

看一个协议是否成熟，要看它是否在修"边界情况"而不是"核心逻辑"。

v1.27.1 修的是：
- auth 预注册的 conformance（边界情况）
- transport error 被静默吞掉（错误处理）
- example URL 打开的命令注入（安全问题）
- HTTP 404 vs 400（API 正确性）

**这些都是"边缘情况"，不是"核心设计有问题"**。当一个协议的 changelog 开始聚焦边界情况时，说明核心已经稳固。

**OpenAI Agents SDK 和 Google ADK 的跟进**（都在 2026 年 3 月同期更新 MCP 支持）说明：大厂在用脚投票——MCP 已经从"Anthropic 的side project"变成了"事实标准"。

### 洞见 3：Sampling 是 MCP 最被低估的能力

大多数讨论 MCP 的文章都聚焦在"Tools"上。但 Sampling（服务端反向调用客户端的 LLM）是一个被低估的东西：

```
它解决的问题：MCP 服务端如何获得 LLM 能力？
传统方案：服务端自己部署一个 LLM API key
MCP 方案：直接用已连接的客户端的 LLM
```

**这意味着**：在 MCP 架构里，LLM 不是被调用的工具，而是可以被共享的资源。这和通信系统里的"网络资源共享"逻辑一致——不是每个节点都独立部署资源，而是按需调用共享资源池。

### 洞见 4：MCP 的安全风险是真实的，但不能因噎废食

MCP 的安全模型（OAuth 2.1 + PKCE）设计得很好，但**实现层面**的问题还很多：
- 社区服务器没有安全审核
- STDIO 模式下信任本地进程（类比：手机允许任意 App 获取 root 权限）
- 服务发现机制缺失导致"连接前无法验资"

**但这不意味着不用 MCP**。就像 HTTPS 早期也有很多实现 bug，但不能因此退回 HTTP。协议层的安全设计是好的，实现层的完善需要时间。

**对 OpenClaw 用户的实际建议**：如果要用 MCP 连接第三方服务，优先选择有明确维护者、有安全审查的企业级服务器。不要随意连接来路不明的社区 MCP 服务器。

### 洞见 5：MCP vs Google Task API = 工具层 vs 任务层，两者是互补的

Google ADK v2.0 的 Task API 和 MCP 解决的是不同层级的问题：

| 层级 | MCP | Google Task API |
|------|-----|-----------------|
| 定位 | 工具集成协议 | Agent 编排协议 |
| 粒度 | 函数调用（tool call） | 任务委托（task delegation） |
| 适用场景 | AI → 外部工具 | Agent → Agent |
| 状态 | 无状态/会话 | 有状态任务追踪 |

**结论**：两者不竞争，会共存。就像通信系统里 SS7（SIP）和 HTTP 同时存在——不同层级用不同的协议。MCP 会成为"AI 系统的 SIP"，Task API 可能会成为"AI 系统的 HTTP/REST"。

## 七、对实际工作的启发

### 7.1 如果要构建 AI 应用

**MCP 的价值**：减少重复造轮子。如果你的应用需要调用 GitHub、Google Drive、数据库——用现成的 MCP 服务器，比自己写集成代码省 80% 的工作量。

**建议路径**：
1. 先用 MCP Registry 看看有没有现成的服务器
2. 用 OpenAI Agents SDK 或类似框架处理 retry、auth、error normalization
3. 自建 MCP 服务器：当没有现成的、需要特定业务逻辑时

### 7.2 如果担心安全

**现状**：企业 MCP 部署最大的风险不是协议本身，而是社区服务器的信任问题。

**应对策略**：
- 企业内部 MCP 服务器必须走 OAuth 授权流程
- 外部社区服务器要做安全 review（类比：npm install 之前先看代码）
- 最小权限原则：scope 只给需要的，不要给 `*`

### 7.3 如果在 RF 测试领域想用 AI Agent

结合我这两周的学习，路径更清晰了：

**第一步**：用 ReAct 架构让 LLM 协调测试仪器（VNA、频谱仪）
**第二步**：用 MCP 标准化仪器控制接口
**第三步**：如果需要多步骤决策，用 Plan-and-Execute 分离战略和战术

**MCP 在 RF 场景的潜力**：
- 仪器厂商可以提供 MCP 服务器，AI 应用商可以即插即用
- 不用每个 AI 平台都写一遍 GPIB/SCPI 驱动
- 但目前仪器厂商的 MCP 服务器生态还不存在——这是未来机会

## 附录：MCP 的四个发展阶段判断

我把 MCP 的发展划分为四个阶段，现在处于第三阶段的早期：

| 阶段 | 特征 | 时间 |
|------|------|------|
| 1. 概念验证 | Anthropic 发布，单一工具，demo 级别 | 2024.11 |
| 2. 开发者采用 | 工具生态形成，TypeScript/Python SDK 成熟 | 2025 |
| 3. 企业早期 | 50+ 企业集成，安全模型落地，但扩展性差 | 2026.Q1 |
| 4. 大规模生产 | 横向扩展、服务发现、企业 SSO → 成为基础设施 | 2026 下半年-2027 |

**现在（2026.03）是阶段 3 的早期**，正在修复生产问题，尚未到达"即插即用"的企业级体验。
